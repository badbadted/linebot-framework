# 伺服器運作實況：傳輸・記錄・處理

> 這份講的是 **這台 LINE bot 伺服器實際在怎麼運作** —— 資料怎麼傳輸、記錄了什麼、請求怎麼被處理。
> 目的：把這套運作模式當作**另一個專案的參考架構**。所有 host/port/路徑/DB/endpoint 都是真實值。
> **不含密鑰**，secret/token 一律環境變數。

---

## 0. 一句話總覽

```
LINE ──(HTTPS)──▶ Cloudflare Tunnel ──▶ Mac mini:3100（Node/Express）
                                              │
                  ┌───────────────────────────┼───────────────────────────┐
                  ▼ 傳輸                       ▼ 記錄                       ▼ 處理
        reply/push 回 LINE          每筆寫 _logs.db            驗簽→去重→比對→執行
        廣播 → PC TTS               業務資料 SQLite/Firestore   router + plugin + provider
```

三支柱：**傳輸**（資料怎麼進出）、**記錄**（記了什麼、存哪）、**處理**（收到後怎麼跑）。

## 2. 傳輸（Transport）

### 2.1 入站：LINE → 伺服器

```
LINE Platform
  → POST https://bot.pushbike-training.app/line/webhook
  → Cloudflare 邊緣（驗 TLS、擋 DDoS）
  → 加密通道 → cloudflared(Mac mini) → http://localhost:3100/line/webhook
  → Express 收到，body 含 events[]，header 含 X-Line-Signature
```

- 協定：對外 HTTPS（CF 終止 TLS），對內 HTTP（localhost）
- 認證：靠 `X-Line-Signature`（HMAC-SHA256，用 channel secret 驗）
- **先回 200** 再處理（LINE 要求快速回應，否則重送）

### 2.2 出站：伺服器 → LINE

呼叫 `https://api.line.me/v2/bot/message/*`，header 帶 `Bearer <channel access token>`：

| 方法 | 何時用 | 程式位置 |
|------|--------|----------|
| `reply` | 回覆使用者訊息（replyToken，30 秒內） | `line-api.js` reply() |
| `push` | 主動推播（排程/事件） | line-api.js push() |
| `multicast` | 多人推播 | line-api.js multicast() |

策略：**能 reply 就 reply（免費），失敗 fallback push**。

### 2.3 出站：伺服器 → 其他服務

| 目標 | 位址 | 用途 |
|------|------|------|
| 家裡 PC TTS | `http://192.168.0.229:9880/tts`（`play:true`） | `/廣播` 語音播報 |
| Luna LLM | `http://localhost:18797`（luna provider） | 未匹配訊息的 AI 兜底回覆 |
| Firestore | Google Cloud（專案 sipangzi003） | 妞揪 niujiu 雲端資料 |

### 2.4 排程／外部觸發（入站，非 LINE）

- 內建 **node-cron**（同進程）定時觸發 → push
- 也開 `POST /api/cron/:jobName`，讓外部系統（雲端 cron、HA automation）打進來觸發排程

### 傳輸特性速查

| 通道 | 方向 | 協定 | 認證 | 備註 |
|------|------|------|------|------|
| LINE webhook | 入 | HTTPS | X-Line-Signature | 先回 200 |
| LINE reply/push | 出 | HTTPS | Bearer token | reply 30 秒一次性 |
| /api/* 管理端點 | 入 | HTTP（經 Tunnel 即 HTTPS） | apiAuth | log/stats/health |
| /api/cron/:job | 入 | 同上 | apiAuth | 外部觸發排程 |
| PC TTS 廣播 | 出 | HTTP（內網） | 無（內網信任） | 25s 逾時 |

## 3. 記錄（Recording / Persistence）

資料分三層存，各司其職：

### 3.1 訊息日誌 — `data/_logs.db`（每筆訊息都記）

**每一筆**進到 webhook 的訊息（不論有沒有命中指令）都會寫一筆 log。表 `message_logs`：

| 欄位 | 內容 |
|------|------|
| `timestamp` | 收到時間（localtime） |
| `user_id` | 發訊者 |
| `source_type` | user / group / room |
| `text` | 訊息原文（截斷 2000 字） |
| `route` | 命中的路由名（沒命中 = `unmatched`） |
| `plugin` | 來源 plugin |
| `type` | query / action / fallback / scope-blocked / group-blocked / ignored |
| `response` | 回覆內容（截斷 500 字） |
| `duration_ms` | 處理耗時 |
| `error` | 錯誤訊息（有才填） |

特性：
- **WAL 模式**、三個索引（timestamp / user_id / route）
- 寫入用 prepared statement，**try/catch 包住**（記 log 失敗不影響主流程）
- 涵蓋所有結果：命中、被 scope/權限擋、fallback、錯誤 —— 都各記一種 `type`

### 3.2 查詢與統計（`/api/logs`、`/api/logs/stats`）

| 端點 | 功能 | 參數 |
|------|------|------|
| `GET /api/logs` | 查訊息記錄 | `limit/offset/userId/from/to/route/errors` |
| `GET /api/logs/stats?days=7` | 統計 | 每日量、指令排行 top10、活躍使用者 top10、錯誤數、未匹配數 |
| `GET /dashboard` | 視覺化儀表板 | 讀上面兩個 API |

**PII 遮蔽**：API 回傳時 `user_id` 只顯示前 8 碼（`maskUserId`），不外洩完整 LINE ID。
這些 `/api/*` 端點受 **apiAuth** 保護（webhook 不受限）。

### 3.3 業務資料 — 每個 plugin 一個 SQLite（isolated）

| 檔案 | 屬於 | 存什麼 |
|------|------|--------|
| `data/bike.db` | 滑步車 | 選手 / 秒數 / 綁定 |
| `data/swim.db` | 游泳 | 同上 + 泳式 |
| `data/todo.db` `data/work.db` `data/travel.db` | 待辦/工作/旅遊 | 各自項目 |
| `data/fitness.db` `data/routines.db` | 健身/例行 | 各自資料 |

- **隔離模式**：每個 plugin 只拿到自己的 DB（`getDatabase(pluginName)`），互不污染
- plugin 在 `init()` 自己建表（`CREATE TABLE IF NOT EXISTS`），升級欄位用 `ALTER` 容錯

### 3.4 雲端資料 — Firestore（niujiu）

妞揪用 Firebase Firestore（專案 `sipangzi003`），collections：events / participants / users / restaurants…
特點：**onSnapshot 即時監聽** → 資料一變就 push 通知（取消/額滿）。

### 記錄分層速查

| 層 | 存哪 | 記什麼 | 保留 |
|----|------|--------|------|
| 訊息日誌 | `_logs.db` | 每筆收發 + 耗時 + 錯誤 | 長期（可自行清理） |
| 業務資料 | `data/<plugin>.db` | 各功能的實體資料 | 永久 |
| 雲端資料 | Firestore | 跨裝置/即時的資料 | 永久（雲端） |

## 4. 處理（Processing）

### 4.1 請求生命週期（webhook 進來後的 8 步）

```
1. res.status(200).end()        立即回，避免 LINE 重送
2. 驗 X-Line-Signature          HMAC-SHA256 + timing-safe，不符丟棄
3. 防重播                       timestamp 超過 5 分鐘 → 丟
4. 去重                         webhookEventId 看過 → 丟
5. 解析 event                   userId / sourceType / groupId / replyToken / text
6. router.match(text)           regex 依序比對 + scope + 群組權限
7. router.execute → handler     產生回覆（字串 / Flex 物件）
8. reply（失敗 fallback push）   + logger.log(整筆結果)
```

每一步的結果（命中/被擋/fallback/錯誤）都會在第 8 步落一筆 log。

### 4.2 路由比對 + scope + 權限

- **比對**：regex 陣列依註冊順序，第一個命中勝出
- **scope**：`all`（都可）/ `private`（限私訊）/ `group`（限群組）
- **群組權限白名單**：群組預設全關，只有 `groups.json` 列出的 plugin 可用；`_system` 不受限
- 被擋的回傳帶 `scopeBlocked` / `groupBlocked`，webhook 端決定回提示或靜默

### 4.3 Provider 注入（外接服務統一管理）

啟動時 `Provider Registry` 依 config 初始化，注入給 plugin：
- `db`：SQLite（isolated 模式，每 plugin 一檔）
- `llm`：Luna（localhost:18797），給未匹配訊息兜底
- 沒設定的 provider = null，plugin 自行判斷降級

### 4.4 Plugin 執行模型

- 啟動掃 `plugins/` 目錄，依白名單（config.enabled）載入
- 每個 plugin：`commands`（→ router）、`schedules`（→ scheduler）、`init(ctx)`（建表/啟 listener/註冊額外路由）
- `init(ctx)` 拿得到 `lineApi / router / scheduler / db / providers`

### 4.5 Fallback 與錯誤處理

- 未命中 + 以 `/` 開頭 → 回「找不到指令」（**不丟 LLM**，避免幻覺假成功）
- 未命中 + 一般文字 → 私訊或有 `_llm` 權限的群組才丟 LLM
- handler 丟錯 → webhook try/catch 記 error log，**不回覆**（已先回 200）
- 記 log 本身丟錯 → 只 console.error，不影響主流程

## 6. 拿來當另一個專案的參考

### 可直接沿用的模式

- **「先回 200 再處理」+ 簽章驗證 + 去重防重播** → 任何 webhook 服務都適用
- **每筆請求落一筆結構化 log（含耗時/結果/錯誤）→ 查詢+統計 API** → 通用可觀測性骨架
- **資料分層**：操作日誌（一張表）／業務資料（隔離 DB）／即時雲端（Firestore）各自分開
- **Provider Registry**：把 DB/LLM/Cache 等外接服務統一注入，沒設定就降級
- **白名單權限 + scope**：多租戶/多群組情境的存取控制

### 換專案要改的

| 元件 | 換成 |
|------|------|
| ingress | 你的 Tunnel / 雲端網址 |
| 簽章驗證 | 你的平台簽章算法（LINE / Slack / Telegram 各不同） |
| 出站 API | 你的訊息平台 send API |
| 業務 plugin | 你的功能 |

### 最小複刻清單

- [ ] webhook：回 200 → 驗簽 → 去重 → 解析
- [ ] logger：一張 `message_logs` 表 + `log()` + `/api/logs` + `/api/logs/stats`（含 PII 遮蔽）
- [ ] router：regex → handler，回字串自動回覆
- [ ] provider 注入：db（至少）
- [ ] health endpoint + stdout log + 重啟腳本
